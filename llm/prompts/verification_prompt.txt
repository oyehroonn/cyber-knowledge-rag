## Task: Vulnerability Verification

Analyze the evidence to determine if this is a genuine security vulnerability. Consider:

### 1. Evidence Analysis
- Does the evidence actually demonstrate the claimed vulnerability?
- Are there alternative explanations for the observed behavior?
- Is the evidence conclusive or circumstantial?
- What additional evidence would strengthen the case?

### 2. Reproducibility Assessment
- Can this be reliably reproduced?
- Are there specific conditions required?
- Is the behavior consistent?

### 3. Impact Verification
- What is the actual security impact?
- Can sensitive data be accessed?
- Can unauthorized actions be performed?
- What is the blast radius?

### 4. False Positive Analysis
- Could this be intended behavior?
- Are there mitigating controls?
- Is the "vulnerability" actually blocked elsewhere?

### 5. Comparison with Known Vulnerabilities
- How does this compare to similar confirmed vulnerabilities?
- Are the patterns consistent?
- What made similar reports valid/invalid?

## Verification Guidelines

### Verdict Definitions
- **confirmed**: Evidence clearly demonstrates a security vulnerability
- **likely**: Strong evidence, but needs additional confirmation
- **unlikely**: Weak evidence, probably not a real vulnerability
- **false_positive**: Evidence shows this is not a vulnerability

### Confidence Levels
- **0.9-1.0**: Conclusive evidence, high certainty
- **0.7-0.9**: Strong evidence, reasonably certain
- **0.5-0.7**: Moderate evidence, somewhat uncertain
- **0.3-0.5**: Weak evidence, significant uncertainty
- **0.0-0.3**: Very weak evidence, mostly uncertain

### Key Questions
1. Can an attacker actually exploit this?
2. Does the evidence show actual security impact?
3. Is this behavior intentional/by design?
4. What would a skeptical security reviewer think?
